# 前台销售

**功能**：实现菜单页面 PosPage.vue
- POS 前台销售界面：零售商品销售界面

---

# 目标

建设一套适用于门店收银场景的前台销售模块，覆盖以下核心能力：
- 商品快速检索与加入购物车
- 顾客识别与会员折扣处理
- 挂单、取单、结算三类核心业务动作
- 单页高密度展示，适合收银台键盘快捷操作

本设计文档仅用于梳理前台销售模块的业务与交互设计，作为后续数据库、接口和页面实现的依据。

# 业务设计

## 1. 业务目标

前台销售用于门店零售收银，要求操作路径短、录入效率高、界面紧凑、支持键盘快捷键，满足以下使用场景：
- 收银员通过扫码枪或商品名称快速搜索商品
- 门店顾客可按会员信息自动带出折扣、余额、积分
- 当前订单可以临时挂起，稍后恢复继续收银
- 结算时支持显示应收、实收、找零等关键信息

## 2. 业务角色

- 操作员：当前登录系统的用户，负责实际收银操作
- 业务员：当前单据归属的销售业务员，默认取系统默认业务员
- 顾客：可为空；为空时按散客处理
- 终端：当前收银终端设备
- 仓库：当前销售商品默认出库仓库

## 3. 业务状态

前台销售主单建议使用以下状态：
- 待结算：单据已创建但尚未正式完成支付
- 已结算：单据完成支付并生效
- 已挂单：当前购物内容被暂存，但不算正式销售单
- 已取消：结算前整单作废

说明：
- 正式销售订单表可仅保存待结算、已结算两种状态
- 挂单建议单独存入挂单表，不直接写正式销售表

## 4. 业务规则

### 4.1 销售单号规则

- 格式：P-YYMMDD + 当前时间毫秒数值
- 示例：P-260427171234567
- 要求唯一，便于人工识别日期

### 4.2 仓库规则

- 仓库必填
- 默认带出默认仓库
- 默认仓库来自 biz_warehouse 的默认仓库标记字段
- 当切换仓库时，新增商品默认使用当前选中仓库
- 若商品行允许单独改仓，则每一行保留自己的仓库信息

### 4.3 业务员规则

- 业务员必填
- 默认带出销售业务员中的默认业务员
- 支持弹框切换

### 4.4 操作员规则

- 操作员必填
- 自动取当前登录用户
- 页面只读显示，不允许手工修改

### 4.5 终端规则

- 终端必填
- 默认带出销售终端中的默认终端
- 支持弹框切换

### 4.6 顾客规则

- 顾客可为空
- 未选择顾客时，按散客处理
- 选择顾客后需展示以下信息：
	- 名称
	- 余额
	- 积分
	- 会员号
	- 会员类型
	- 折扣率

### 4.7 商品规则

- 商品来源于 biz_goods
- 搜索方式支持：
	- 商品名称模糊查询
	- 商品编码查询
	- 条形码或其他编码查询
- 商品加入购物车后，需保留快照信息，避免后续商品资料修改影响历史单据

### 4.8 折扣规则

折后价根据顾客会员折扣与商品是否参与会员折扣共同决定。

建议规则如下：
- 若未选择顾客，则折扣率 = 1
- 若顾客无会员类型，则折扣率 = 1
- 若商品不参与会员折扣，则该商品折扣率 = 1
- 若商品参与会员折扣，则折扣率 = 顾客会员类型的 discount 字段值

计算方式：
- 单行原价小计 = 销售价 × 数量
- 单行折后单价 = 销售价 × 折扣率
- 单行折后小计 = 单行折后单价 × 数量
- 总金额 = 所有单行原价小计汇总
- 折后价 = 所有单行折后小计汇总
- 优惠金额 = 总金额 - 折后价

说明：
- biz_member_type.discount 若为 0.9，表示 9 折
- 页面展示时可显示为 90% 或 9折，前后端统一即可

### 4.9 积分规则

- 仅商品参与积分时才累计积分
- 仅已结算订单累计积分
- 建议积分规则：按折后金额向下取整累计积分
- 散客不累计积分

### 4.10 付款规则

- 付款方式默认现金
- 支持扩展的付款方式建议包括：
	- 现金
	- 微信
	- 支付宝
	- 银行卡
	- 余额
- 支付金额由操作员输入
- 应找金额 = 支付金额 - 折后价
- 若应找金额 < 0，表示仍有未收金额，应阻止完成结算

### 4.11 挂单规则

- 挂单仅保存当前购物车状态，不扣库存，不生成正式销售记录
- 挂单成功后清空当前页面购物车，并重新生成新的销售单号
- 挂单建议按操作员隔离，只展示当前操作员自己的挂单

### 4.12 取单规则

- 取单时展示当前操作员的挂单列表
- 选择一条挂单后，恢复购物车、顾客、仓库、业务员、终端等上下文
- 成功取单后，该挂单记录删除或标记已取走

### 4.13 结算规则

- 结算前必须至少存在一条商品明细
- 结算时校验支付金额必须大于等于折后价
- 结算成功后：
	- 写入销售主单
	- 写入销售明细
	- 扣减库存
	- 更新积分
	- 若付款方式为余额，扣减顾客余额
- 结算完成后清空购物车并生成下一笔销售单号

# 实现

## step 1: 在 pg 目录下，设计【前台销售】相关数据表

### 1）修改仓库表

现有 biz_warehouse 表中建议增加字段：
- is_default：是否默认仓库，1=是，0=否

初始化规则：
- 将“北京总仓”设置为默认仓库
- 其他仓库设为非默认
- 同时建议增加唯一约束策略，保证仅有一个默认仓库

### 2）设计前台销售主表

建议表名：biz_pos_order

建议字段：

| 字段名 | 说明 |
|---|---|
| id | 主键 |
| sale_no | 销售单号，唯一 |
| warehouse_id | 仓库ID |
| sales_staff_id | 销售业务员ID |
| operator_id | 操作员ID |
| terminal_id | 终端ID |
| customer_id | 顾客ID，可空 |
| goods_count | 商品总数 |
| total_amount | 总金额 |
| discount_amount | 优惠金额 |
| payable_amount | 折后价/应付金额 |
| payment_method | 付款方式 |
| paid_amount | 支付金额 |
| change_amount | 应找金额 |
| status | 状态：待结算、已结算 |
| remark | 备注 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

设计说明：
- 主表保存订单汇总信息
- 商品明细不建议以 JSON 直接存主表，应拆分子表，便于统计、审计和后续扩展

### 3）设计前台销售明细表

建议表名：biz_pos_order_item

建议字段：

| 字段名 | 说明 |
|---|---|
| id | 主键 |
| order_id | 关联销售主表 |
| goods_id | 商品ID |
| goods_name_snapshot | 商品名称快照 |
| unit_name_snapshot | 单位名称快照 |
| quantity | 数量 |
| sale_price | 原价 |
| discount_rate | 折扣率 |
| final_price | 折后单价 |
| final_amount | 折后小计 |
| warehouse_id | 仓库ID |
| enable_points | 是否参与积分 |
| enable_discount | 是否参与折扣 |
| created_at | 创建时间 |

设计说明：
- 使用快照字段保存单据生成时的商品信息
- 每个商品行保留仓库信息，便于后续支持跨仓出货

### 4）设计挂单表

建议表名：biz_pos_hold_order

建议字段：

| 字段名 | 说明 |
|---|---|
| id | 主键 |
| hold_no | 挂单号 |
| operator_id | 操作员ID |
| warehouse_id | 仓库ID |
| sales_staff_id | 销售业务员ID |
| terminal_id | 终端ID |
| customer_id | 顾客ID，可空 |
| goods_count | 商品总数 |
| total_amount | 总金额 |
| payable_amount | 折后价 |
| cart_json | 当前购物车JSON快照 |
| remark | 备注 |
| created_at | 创建时间 |

设计说明：
- 挂单表用于临时存储前台购物车状态
- cart_json 用于保存完整的页面临时上下文，降低恢复成本
- goods_count、total_amount、payable_amount 冗余存储，用于列表快速展示

### 5）设计建议

- 销售正式单与挂单分表存储
- 正式销售单据必须支持事务提交
- 所有金额字段统一使用 NUMERIC
- 所有状态字段统一使用 SMALLINT 或 VARCHAR 枚举，避免语义不清

## step 2: 后端 jxcServer 工程，创建【前台销售】相关 API

参考前台销售业务说明，建议设计以下接口。

### 1）基础初始化接口

接口建议：
- GET /api/pos/bootstrap

用途：
- 返回默认仓库
- 返回默认业务员
- 返回默认终端
- 返回当前登录用户信息

### 2）商品搜索接口

接口建议：
- GET /api/pos/goods/search

查询参数：
- keyword：商品名称/编码/条码关键字
- warehouse_id：当前仓库ID

返回建议字段：
- 商品ID
- 商品编码
- 商品名称
- 单位
- 销售价
- 当前仓库库存
- 是否参与积分
- 是否参与折扣

### 3）挂单保存接口

接口建议：
- POST /api/pos/hold

用途：
- 保存当前购物车内容到挂单表

### 4）挂单列表接口

接口建议：
- GET /api/pos/hold/list

用途：
- 查询当前操作员的挂单列表

### 5）取单接口

接口建议：
- POST /api/pos/hold/take

用途：
- 根据挂单ID取回挂单数据
- 取单成功后删除或更新挂单状态

### 6）删除挂单接口

接口建议：
- DELETE /api/pos/hold/delete

用途：
- 删除不再需要的挂单

### 7）结算接口

接口建议：
- POST /api/pos/checkout

用途：
- 创建销售主单
- 创建销售明细
- 扣减库存
- 更新会员积分/余额

### 8）建议的后端事务范围

结算接口中以下动作必须放在同一事务内：
- 新增销售主单
- 新增销售明细
- 扣减库存
- 更新顾客积分
- 更新顾客余额

若其中任一步失败，必须整体回滚。

## step 3: 前端 jxcClient 实现 “前台销售” 菜单页面

### 页面总体要求

- 一页展示
- 不要上下滚动
- 字体设计为小号
- 以键盘操作优先，鼠标操作为辅助
- 页面进入后默认聚焦商品搜索输入框

### 页面建议布局

整体分为：
- 上区域
- 中区域
- 下区域

页面高度建议采用：
- 上区域：约 80px 至 100px
- 中区域：占据主要空间
- 下区域：约 140px 至 180px

# 页面交互补充说明

## 快捷键建议

- Enter：搜索或添加商品
- Up：上移选中项
- Down：下移选中项
- Delete：删除选中商品
- F1：修改数量
- F4：选择顾客
- F8：挂单
- F9：取单
- F12：整单清空
- +：打开结算弹框
- Esc：关闭当前弹框

## 焦点流转建议

- 页面进入默认焦点在商品搜索输入框
- 搜索回车后焦点进入搜索结果列表
- 搜索结果回车添加商品后，焦点返回搜索输入框
- 弹框关闭后，焦点回到主搜索输入框

## 页面提示建议

需要统一的轻提示或消息提示，覆盖以下场景：
- 商品不存在
- 库存不足
- 挂单成功
