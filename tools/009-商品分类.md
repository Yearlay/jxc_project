# 商品分类

**功能**：实现二级菜单页面
- 商品管理 → 商品分类（菜单 id=22，path=/goods/category）

---
# 输入

### 商品分类
1）字段说明：`name`（分类名称）、`parent_id`（父分类 ID，0 表示一级分类）
2）默认数据：预置“未分类”为系统默认分类，不允许删除、不允许修改名称、不允许新增子分类
3）预置一级分类：`食品`、`家具`、`五金`、`饰品`
4）业务说明：新增、修改、删除
5）不设计状态字段（启用/禁用）
6）分类树最多支持 3 级：一级分类 + 二级分类 + 三级分类，不支持四级及以上

### 规则约束
1）同一父分类下，分类名称不允许重复
2）删除一级分类前，若存在子分类，则禁止删除并提示“请先删除子分类”
3）“未分类”是系统保留分类：不支持增加、修改、删除三点操作
4）新增子分类时，不允许挂到“未分类”下面
5）三级分类下不允许继续新增子分类

# 实现

## step 1: 在 `pg` 目录下，新建商品分类表结构 SQL
新建文件建议：`11_create_goods_category_table.sql`

表名：`goods_category`

字段建议：
- `id SERIAL PRIMARY KEY`
- `name VARCHAR(50) NOT NULL`
- `parent_id INTEGER NOT NULL DEFAULT 0`
- `sort INTEGER NOT NULL DEFAULT 0`
- `is_system SMALLINT NOT NULL DEFAULT 0`：是否系统保留分类，`1=是`、`0=否`
- `created_at TIMESTAMP NOT NULL DEFAULT NOW()`

约束建议：
- `(parent_id, name)` 唯一，避免同级重名
- `parent_id >= 0`

初始化数据：
1）插入默认分类：`未分类`，`parent_id=0`，`is_system=1`
2）插入一级分类：`食品`、`家具`、`五金`、`饰品`，`parent_id=0`

同步更新：
1）在 `00_reinit_db.sh` 中追加执行 `11_create_goods_category_table.sql`
2）step 编号顺延到下一个可用序号

## step 2: 后端 `jxcServer` 工程，创建商品分类 API
建议新建文件：`api/goods_category.py`

Blueprint：`goods_category_bp`

接口设计：

### 1. 获取分类树
- `GET /api/goods-category/tree`
- 返回树形结构，前端直接用于 `a-tree`
- 返回字段至少包含：`id`、`name`、`parent_id`、`is_system`、`children`

### 2. 新增分类
- `POST /api/goods-category/add`
- Body：`{ name, parent_id }`
- 规则：
	- `name` 不能为空
	- `parent_id` 必填，`0` 表示新增一级分类
	- 若 `parent_id != 0`，父分类必须存在
	- 不允许在“未分类”下新增子分类
	- 父分类若已经是三级分类，则不允许继续新增下级
	- 同一父分类下名称不能重复

### 3. 修改分类
- `PUT /api/goods-category/update`
- Body：`{ id, name }`
- 规则：
	- `id` 必填
	- `name` 不能为空
	- “未分类”不允许修改
	- 同一父分类下名称不能重复

### 4. 删除分类
- `DELETE /api/goods-category/delete`
- Body：`{ id }`
- 规则：
	- `id` 必填
	- “未分类”不允许删除
	- 若当前分类下存在子分类，则禁止删除

后端实现要求：
1）统一使用现有返回格式：`{ code, msg, data }`
2）异常时返回明确提示，不要直接抛原始数据库错误
3）在 `app.py` 中注册 blueprint：
	 - `from api.goods_category import goods_category_bp`
	 - `app.register_blueprint(goods_category_bp, url_prefix='/api')`

完成后验证：
1）查询分类树
2）新增一级分类
3）新增二级分类
4）新增三级分类
5）修改分类名称
6）验证同级重名拦截
7）验证“未分类”不可改不可删
8）验证有子分类的一级/二级分类不可删
9）验证三级分类下不可继续新增子分类
10）删除测试分类

## step 3: 前端 `jxcClient` 实现“商品分类”页面
页面建议：`src/views/GoodsCategoryManage.vue`

API 文件建议：`src/api/goodsCategory.js`

在 `Home.vue` 的 `COMPONENT_MAP` 中注册：

```js
'/goods/category': GoodsCategoryManage,
```

页面交互要求：
1）整体使用 `a-tree` 展示分类树
2）节点右侧展示三点操作按钮
3）操作项包含：新增、修改、删除
4）“未分类”节点不显示任何三点操作
5）一级分类支持：新增子分类、修改、删除
6）二级分类支持：新增子分类、修改、删除
7）三级分类支持：修改、删除；不再继续新增下级分类

建议 UI 布局：
1）顶部标题：商品分类
2）右上角全局按钮：新增一级分类
3）中间卡片区域显示树形分类
4）节点内容左侧是分类名称，右侧是操作菜单

弹窗设计：

### 新增分类弹窗
- 字段：`分类名称`、`父分类`
- 若是点击“新增一级分类”，则 `parent_id=0` 并隐藏父分类选择
- 若是点击某一级分类的“新增”，则自动带入该分类作为父分类，且不可改

### 修改分类弹窗
- 字段：`分类名称`
- 展示当前父分类名称，仅作只读说明

### 删除确认弹窗
- 二次确认：`确定要删除分类“xxx”吗？`
- 若后端返回有子分类不可删，前端直接提示后端消息

树节点展示建议：
1）系统默认分类“未分类”使用 `a-tag` 或浅色标识体现“系统默认”
2）一级分类标题加粗
3）二级分类、三级分类使用递进缩进，保持层级清晰

前端逻辑要求：
1）页面初始化加载分类树
2）新增/修改/删除成功后，重新拉取分类树
3）弹窗关闭后重置表单
4）删除前不在前端自行猜测是否可删，以后端校验为准
